## 2. Web Interface: Admin vs Technician Sides
The existing project already has `AuthContext.jsx` and `backend/app/routers/auth.py` with technician administration routes. We can extend this with role-based access:
- **Admin role**: Full system configuration, user/technician management, access to all AI reports, system history, and raw data
- **Technician role**: Access to assigned devices only, error alerts with localization details, action buttons, and troubleshooting guides

No major new architecture needed, just extend existing auth and frontend routing to check user roles before rendering components.

## 3. IoT Simulation with Node-RED
Node-RED will replace/supplement the existing ESP32 simulators. It fits the current data flow directly:
- Node-RED publishes simulated sensor data to the same MQTT broker (port 1883) using the existing topic contract `cooperative/device/{device_id}/sensor/{sensor_id}`
- The existing `backend/app/mqtt_subscriber.py` will pick up these readings automatically with no backend changes
- Node-RED can simulate realistic sensor patterns and inject fault scenarios (e.g., gradually rising temperature, sudden vibration spike) for testing and defense demos
- Teacher's note: Node-RED also provides built-in real-time dashboards to show live machine/sensor state alongside our existing React dashboard

## 4. AI Error Detection & Localization
This is the core innovative component. No pre-trained external model is needed; we can train models on our own simulated data:

### Recommended Approach (per teacher's suggestions)
1. **Anomaly Detection**: Use Isolation Forest (lightweight, unsupervised, teacher-recommended) to flag abnormal readings without needing labeled fault data upfront
2. **Error Classification & Localization**: Train a Random Forest classifier (teacher-recommended) on labeled data generated by Node-RED (normal vs. high temp, low flow, high vibration, etc.) to:
   - Classify the exact error type
   - Map errors to physical components using a predefined sensor-to-component mapping (e.g., temperature sensor 3 maps to oven heating element)
   - Use Random Forest feature importance to show which sensors contributed most to the error, confirming the fault location

### Implementation Details
- Use scikit-learn (Python) to implement models, integrated into the existing FastAPI backend as a new module
- Models run inference on new sensor readings (either real-time via MQTT subscriber callback or batched every 10 seconds)
- Store inference results (error type, confidence score, component location) in PostgreSQL alongside sensor readings

### Open Questions
- Supervised (Random Forest with labeled data) vs. unsupervised (Isolation Forest only)? Labeled data can be generated easily via Node-RED fault injection
- Where to deploy inference: real-time on each reading (low latency) or batched (lower load)?

### Teacher's Comparison Requirement
Run a simple custom rule-based algorithm (e.g., threshold checks) in parallel with the AI model. Compare precision, recall, and false positive rates to demonstrate the AI's value during the defense.
